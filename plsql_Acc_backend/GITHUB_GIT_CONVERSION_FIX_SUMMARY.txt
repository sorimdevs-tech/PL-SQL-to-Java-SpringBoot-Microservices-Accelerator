FAILURE SYMPTOM:

  Error Type: Semantic Validation (after 3 repair attempts)
  
  Error Messages:
    - "ViewitemLibraryService.java calls row.getVideoid() but BookEntity does not define it"
    - "HandlereturnsLibraryService.java calls row.getCardid() but BookEntity does not define it"
    - "HandlereturnsLibraryService.java calls row.getItemid() but BookEntity does not define it"
    - "RemoveitemLibraryService.java calls row.getVideoid() but BookEntity does not define it"

CHAIN OF CAUSALITY:
  1. PL/SQL Source Analysis
     └─> Identifies 16 procedures across 8 entities (BOOK, VIDEO, CARD, etc.)
  
  2. Entity Generation
     └─> Creates 8 JPA entity classes (BookEntity, VideoEntity, CardEntity, ...)
     └─> Each entity has only its own fields (BookEntity ≠ has Videoid field)
  
  3. Service Generation (Deterministic)
     └─> _generate_deterministic_service_from_unit() creates hardcoded logic
     └─> Assumes all related data accessible via "driving" entity
     └─> Uses BookEntity 'row' variable for ALL logic (video + card + book)
     └─> Generates code like: row.getVideoid() [WRONG - BookEntity doesn't have it]
  
  4. Semantic Validation (Attempt 1)
     └─> Detects invalid getter calls on wrong entity types
     └─> Returns 5 semantic issues
  
  5. Service Repair Attempt (Attempt 1)
     └─> LLM repair prompt called with validation feedback
     └─> BUT: Prompt was too generic (only showed first 5 issues)
     └─> NO entity field mapping context
     └─> NO variable-to-entity type information
     └─> LLM makes generic fixes that don't address core problem
  
  6. Validation Retry (Attempt 2 & 3)
     └─> Same errors persist because LLM lacked context
     └─> Pipeline fails after 3 attempts (max_repair_loops=3)

ROOT PROBLEM: **Entity Context Blindness in LLM Repair**
  - Services access wrong entity types
  - Repair prompt doesn't explain which variable should be which type
  - LLM can't generate correct fixes without this semantic mapping

================================================================================
SOLUTION: DYNAMIC ENTITY MAPPING (DYN-FIX-8 & DYN-FIX-9)
================================================================================

PHILOSOPHY: 
  Make the LLM "entity-aware" by providing complete semantic context about
  which entity types exist, which fields they have, and which variables should
  reference which types.

TWO-PART FIX:

DYN-FIX-8: Entity-Variable Type Tracking
==========================================

NEW METHOD: _extract_variable_entity_types(service_code, entities)
LOCATION: src/converter/llm_engine.py, lines 3507-3525

PURPOSE:
  Parse service code to determine which entity type each variable holds

ALGORITHM:
  1. Extract entity class names from files (BookEntity.java → BookEntity)
  2. Find declarations: EntityType variable = ...
  3. Build mapping: {variable_name → EntityClassName}

EXAMPLE OUTPUT:
  Service Code:
    BookEntity row = bookRepository.findById(1L).orElseThrow();
    VideoEntity video = videoRepository.findById(2L).orElseThrow();
  
  Extracted Mapping:
    {'row': 'BookEntity', 'video': 'VideoEntity'}

DYN-FIX-9: Semantic Issue Analysis
===================================

ENHANCEMENT TO: _build_service_repair_prompt()
LOCATION: src/converter/llm_engine.py, lines 3517-3600

PURPOSE:
  Parse validation errors to infer which entity types are needed and
  provide explicit recommendations to the LLM

ALGORITHM:
  1. Scan error messages: "calls row.getVideoid() but BookEntity"
  2. Extract: variable='row', field='Videoid', current_entity='BookEntity'
  3. Cross-check all entities: "Does VideoEntity have Videoid? YES!"
  4. Build recommendations: 
     "VideoEntity is needed by: ['row']"
     "It provides fields: ['videoid', 'title', 'duration']"

PROMPT ENHANCEMENT:
  NEW SECTIONS ADDED:
    - "ENTITY FIELD MAPPINGS (DYN-FIX-7)"
      Shows all available fields on each entity type
    
    - "ENTITY-VARIABLE TYPE BINDINGS (DYN-FIX-8)"
      Shows which variable holds which type
    
    - "ISSUES SUGGEST THESE CHANGES (DYN-FIX-9)"
      Shows which entity types are needed and why

  NEW REPAIR RULES:
    - "CRITICAL: For each variable, verify it holds the correct entity type"
    - "You can ONLY call methods on variables for fields in that type"
    - "When fixing field access errors, change variable assignments"
