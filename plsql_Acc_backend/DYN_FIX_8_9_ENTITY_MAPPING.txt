DYNAMIC FIX 8 & 9: ENTITY-VARIABLE TYPE TRACKING AND SEMANTIC REPAIR
================================================================================

PROBLEM SUMMARY
===============
The git conversion was failing with semantic validation errors indicating that 
services were trying to access fields that don't exist on the entity types being used.

ERROR PATTERN (example): 
  "ViewitemLibraryService.java calls row.getVideoid() but BookEntity does not define it"

ROOT CAUSE:
-----------
1. The deterministic service generator (_generate_deterministic_service_from_unit) 
   creates hardcoded logic that assumes all related fields can be accessed via 
   a single "driving" entity type.

2. When multiple entities are involved (BOOK, VIDEO, CARD), the code tries to 
   access fields from all of them through a BookEntity variable named 'row'.

3. The LLM-based repair mechanism (when validation fails) wasn't getting enough 
   semantic context about which variable should hold which entity type.

SOLUTION: DYNAMIC ENTITY MAPPING (DYN-FIX-8 & DYN-FIX-9)
========================================================

DYN-FIX-8: Entity-Variable Type Tracker
----------------------------------------
NEW METHOD: _extract_variable_entity_types(service_code, entities)
LOCATION: src/converter/llm_engine.py, lines 3507-3525

PURPOSE: 
  Extract which entity type each variable is declared as in the service code
  (e.g., detect that variable 'row' is of type BookEntity, 'video' is VideoEntity)

ALGORITHM:
  1. Parse all entity class names from files: BookEntity.java → BookEntity, etc.
  2. Search service code for variable declarations like "BookEntity row = ..."
  3. Build dictionary: {variable_name → EntityClassName}
     Example: {'row': 'BookEntity', 'video': 'VideoEntity', 'card': 'CardEntity'}

OUTPUT USAGE:
  This mapping is injected into the LLM repair prompt so the LLM understands
  what type each variable holds and which methods it can call.

DYN-FIX-9: Semantic Issue Analysis & Recommendations
-----------------------------------------------------
ENHANCEMENT TO: _build_service_repair_prompt()
LOCATION: src/converter/llm_engine.py, lines 3517-3600

PURPOSE:
  Parse validation error messages to infer which entity types need to be used
  and provide explicit recommendations in the LLM prompt.

ALGORITHM:
  1. Scan each validation error message for patterns like:
     "calls row.getVideoid() but BookEntity does not define it"
     
  2. Extract: variable='row', field='Videoid', entity='BookEntity'
  
  3. Check all entities to find which one HAS this field:
     If VideoEntity has 'videoid' field → recommend changing 'row' to VideoEntity
     
  4. Build entity_needs mapping: {EntityType → {variables that should use it}}
     Example: {'VideoEntity': {'row'}, 'CardEntity': {'card'}}

PROMPT INJECTION (NEW SECTIONS):
  - "ENTITY-VARIABLE TYPE BINDINGS (DYN-FIX-8)"
    Shows current variable declarations and which fields each type provides
    
  - "ISSUES SUGGEST THESE CHANGES (DYN-FIX-9)"
    Shows which entity types are needed and which variables should use them
    Example: "VideoEntity is needed by: row, video"
             "It provides fields: videoid, title, duration, ..."

REPAIR RULES (ENHANCED):
  - "CRITICAL: For each variable (row, entity, target, etc.), verify it holds 
    the correct entity type."
  - "You can ONLY call getters/setters on variables for fields that exist on 
    that entity type."
  - "If error says 'calls row.getVideoid() but BookEntity does not define it':
    → Check: what type is 'row'? Does VideoId exist in that type?
    → Fix: Either declare a VideoEntity variable, or adjust assignment"

DATA FLOW
=========

Stage 1: Service Generation
  _generate_deterministic_service_from_unit(unit, entities, repositories, ...)
    └─> Creates initial service code with entity declarations

Stage 2: Semantic Validation
  SemanticValidator.validate(services, entities, repositories, ...)
    └─> Detects invalid field access (e.g., row.getVideoid() on BookEntity)
    └─> Returns list of SemanticIssue objects with detailed messages

Stage 3: Entity Mapping Extrac (NEW)
  _extract_variable_entity_types(service_code, entities)
    └─> Identifies which variables hold which entity types
    
Stage 4: Issue Analysis (NEW)
  _build_service_repair_prompt() parses errors + entity_needs
    └─> Builds comprehensive repair guidance for LLM
    
Stage 5: LLM Repair (ENHANCED)
  repair_provider.generate_code(enhanced_prompt)
    └─> LLM gets complete semantic context
    └─> Generates corrected service code with proper entity types
    
Stage 6: Validation Retry
  SemanticValidator.validate(repaired_services, entities, repositories, ...)
    └─> If successful, repaired code accepted
    └─> If fails again, process repeats (max 3 attempts)

VALIDATION PASSES ✓

TECHNICAL DETAILS
=================

1. Entity Field Extraction (_extract_entity_field_mapping):
   - Uses regex: r'private\s+([a-zA-Z<>?,\s]+?)\s+(\w+)\s*;'
   - Extracts all private fields from compiled entity classes
   - Maps field names to types for validation

2. Variable-Type Tracking (_extract_variable_entity_types):
   - Recognizes entity class names from filenames (BookEntity.java → BookEntity)
   - Matches variable declarations: r'{EntityName}\s+(\w+)\s*='
   - Handles all Java variable declaration syntaxes

3. Issue Analysis (in _build_service_repair_prompt):
   - Parses error messages with: r'calls (\w+)\.get(\w+)\(\) but (\w+) does not define it'
   - Cross-references all entity field mappings
   - Builds recommendations for variable-to-entity assignment

4. Prompt Injection:
   - All entity field info is aggregated into readable sections
   - Issue suggestions are explicit and actionable
   - Repair rules emphasize entity type correctness
