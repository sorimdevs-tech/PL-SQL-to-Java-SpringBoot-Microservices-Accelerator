PROBLEM STATEMENT
=============================================================================

The SQL schema extraction system had two critical issues:

1. MISSING FOREIGN KEY CONSTRAINTS: Only 1 FK detected when 9+ should exist
   - System only parsed FOREIGN KEY from CREATE TABLE statements
   - Ignored ALTER TABLE ... ADD CONSTRAINT ... FOREIGN KEY statements
   
2. INCORRECT ASSUMPTIONS ABOUT COLUMN NAMES:
   - System assumed FK column name == referenced PK column name
   - Failed when: CUSTOMER_ID -> CUSTOMERS.ID (column names mismatch)

=============================================================================
ROOT CAUSE ANALYSIS
=============================================================================

Backend Issues:
1. _extract_alter_table_foreign_keys() existed but had a RIGID regex pattern
   - Pattern: r'alter\s+table\s+([...].*?foreign\s+key...'
   - Failed to match: ADD CONSTRAINT fk_name FOREIGN KEY (...)
   - Didn't handle multi-line formatting properly
   
2. ALTER TABLE FKs were extracted but NOT merged into table definitions
   - FK data in relationships list but missing from table.foreign_keys
   - Inconsistency between UI diagram (relationships) and table details
   
3. No FK extraction in infer_tables_from_dml() fallback path
   - When DDL not available, BMode inferred tables but skipped ALTER TABLE FKs
   
Frontend:
- Frontend ALREADY data-driven (no inferences)
- Renders ONLY relationships provided by backend
- No filtering based on column name assumptions

=============================================================================
SOLUTION IMPLEMENTED (FULLY DYNAMIC)
=============================================================================

BACKEND CHANGES:

1. DYN-FK-1: Improved _extract_alter_table_foreign_keys() regex
   
   BEFORE:
   - Single rigid pattern: r'alter\s+table....*?foreign\s+key...'
   - Failed on ADD CONSTRAINT syntax
   - Non-greedy .*? ineffective with DOTALL flag
   
   AFTER:
   - Pattern 1: Handles "ALTER TABLE ... ADD CONSTRAINT name FOREIGN KEY"
   - Pattern 2: Handles "CONSTRAINT ... FOREIGN KEY" (finds table context dynamically)
   - Supports multi-line formatting with proper whitespace handling
   - Dynamically backtracks to find owning table when needed
   
   Changed lines: 592-678
   Function: _extract_alter_table_foreign_keys(sql_text: str)
   
2. DYN-FK-2: Enhanced CREATE TABLE FK pattern
   
   BEFORE:
   - Basic pattern: r"\bforeign\s+key\s*\(([^)]+)\)..."
   - Didn't capture optional constraint names
   - Missed some valid FK statements
   
   AFTER:
   - Enhanced pattern: Handles optional CONSTRAINT clause
   - Proper handling of constraint names and whitespace
   
   Changed lines: 326-343
   Function: _extract_table_definitions() within loop

3. DYN-FK-3: Merge ALTER TABLE FKs into table definitions
   
   BEFORE:
   - ALTER TABLE FKs only added to relationships list
   - Not merged into table.foreign_keys array
   - Inconsistency between table details and diagram
   
   AFTER:
   - Extract ALTER TABLE FKs
   - Merge into corresponding table.foreign_keys arrays
   - Deduplication at merge point
   - Ensures consistency across all outputs
   
   Changed lines: 4047-4067
   Function: build_discovery_model()
   New logic:
   - Call _extract_alter_table_foreign_keys()
   - Iterate through results
   - Add to table.foreign_keys with deduplication
   - Then use merged data for relationships

4. DYN-FK-4: Extract ALTER TABLE FKs in inference mode
   
   BEFORE:
   - infer_tables_from_dml() skipped ALTER TABLE FKs
   - Only returned inferred FKs from parameter analysis
   
   AFTER:
   - Also calls _extract_alter_table_foreign_keys()
   - Merges extracted FKs into inferred table definitions
   - Ensures completeness whether using DDL or DML inference
   
   Changed lines: 593-609
   Function: infer_tables_from_dml()
   New logic:
   - Extract ALTER TABLE FKs
   - Merge with implied FKs
   - Deduplication for both sources

FRONTEND:
- NO CHANGES REQUIRED
- Already fully data-driven
- Renders only backend-provided relationships
- No assumptions about column name matching

=============================================================================

Before:

After:

  ADD CONSTRAINT FK_INVOICES_CUST

  REFERENCES CUSTOMERS(customer_id)

  source_column: "CUST_ID"
  target_table: "CUSTOMERS"

  {
    "source_column": "CUST_ID",
    "target_table": "CUSTOMERS",
    "target_column": "CUSTOMER_ID"
  }

Frontend renders as:
  INVOICES.cust_id -> CUSTOMERS(customer_id)  ✓ CORRECT

No assumptions, no name matching, pure parser-extracted data.
